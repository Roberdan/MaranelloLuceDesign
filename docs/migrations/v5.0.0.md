# Migration Guide — v5.0.0

> **Severity:** MINOR (semver major for accumulated breaking changes)
> **Affects:** JS APIs (StateScaffold, PanelOrchestrator), CSS entrypoint, Sugar theme specificity
> **Migration effort:** Low to Medium — most changes are search-replace or additive

## Summary

v5.0.0 consolidates the Presentation Runtime introduced in v4.x with four breaking changes:
StateScaffold gains a dedicated `ready` state (replacing the overloaded `partial`),
PanelOrchestrator integrates with AppShellController for layout-aware panel placement,
the `./css` entrypoint now ships the complete design system, and the Sugar theme drops 93%
of its `!important` declarations in favor of higher-specificity selectors. These changes
affect consumers who use StateScaffold's `partial` state as a success indicator, construct
PanelOrchestrator without an AppShell reference, rely on a minimal CSS bundle from `./css`,
or depend on Sugar theme specificity for custom overrides.

## Version Comparison: v4.20.0 to v5.0.0

| Area | v4.20.0 | v5.0.0 |
|---|---|---|
| StateScaffold states | 5 (`loading`, `empty`, `error`, `partial`, `no-results`) | 6 (adds `ready`; `partial` semantics narrowed) |
| PanelOrchestrator constructor | `(registry, navigation)` | `(registry, navigation, shell?)` — optional AppShell integration |

| Sugar `!important` count | 128 declarations | 9 declarations (93% reduction) |
| Minimum Node.js | 18 | 18 (unchanged) |

## Breaking Changes

### [BC-1] StateScaffold: new `ready` state, `partial` semantics changed

**Why it changed:** `partial` was being used as the success/loaded state (e.g., in
DashboardRenderer after data fetch), creating semantic confusion. "Partial" implies degraded
content, not fully loaded. A dedicated `ready` state makes intent explicit.

**Who is affected:** Anyone calling `setState('partial')` to indicate successful data load.

**Before (v4.20.0):**
```ts
// After successful data fetch, show content:
scaffold.setState('partial');

// ScaffoldState type:
type ScaffoldState = 'loading' | 'empty' | 'error' | 'partial' | 'no-results';
```

**After (v5.0.0):**
```ts
// After successful data fetch, show content:
scaffold.setState('ready');

// ScaffoldState type:
type ScaffoldState = 'loading' | 'empty' | 'error' | 'partial' | 'no-results' | 'ready';

// 'partial' is now reserved for degraded content (some data unavailable):
scaffold.setState('partial', 'Some metrics failed to load');
```

**Migrate:**
1. Find all calls to `setState('partial')` that mean "data loaded successfully."
2. Replace with `setState('ready')`.
3. Keep `setState('partial')` only where content is genuinely degraded (some data missing).

---

### [BC-2] PanelOrchestrator: now integrates with AppShellController

**Why it changed:** PanelOrchestrator created containers directly on `document.body`, bypassing
AppShellController's slot system. This caused layout inconsistencies when both were used together
(panels rendered outside the grid, z-index conflicts, broken responsive collapse).

**Who is affected:** Anyone using PanelOrchestrator alongside an AppShell layout.

**Before (v4.20.0):**
```ts
import { PanelOrchestrator } from 'maranello-luce-design-business';

// Two-argument constructor, always creates standalone containers:
const orch = new PanelOrchestrator(registry, navigation);
orch.open('detail', 'side-panel'); // appends to document.body
```

**After (v5.0.0):**
```ts
import { PanelOrchestrator, AppShellController } from 'maranello-luce-design-business';

const shell = new AppShellController(container, { layout: 'side-detail' });

// Three-argument constructor — shell is optional:
const orch = new PanelOrchestrator(registry, navigation, shell);
orch.open('detail', 'side-panel'); // uses shell's detail slot

// Without shell, backward-compatible standalone mode:
const orchStandalone = new PanelOrchestrator(registry, navigation);
```

**Migrate:**
1. If you use PanelOrchestrator with an AppShell, pass the `AppShellController` instance
   as the third constructor argument.
2. If you use PanelOrchestrator without AppShell, no change needed — the third argument
   is optional and defaults to standalone mode.

---

### [BC-3] CSS entrypoint: `./css` export now includes full system

**Why it changed:** `index.css` was missing Sugar theme files, responsive overrides, runtime
layout CSS, and integration styles. Consumers importing `./css` got an incomplete system
that broke Sugar theme rendering, mobile layouts, and runtime components.

**Who is affected:** Anyone importing `maranello-luce-design-business/css` who depends on
the smaller bundle size or has custom CSS load ordering.

**Before (v4.20.0):**
```css
/* index.css included ~105 files, ~250KB */
@import 'maranello-luce-design-business/css';
/* Missing: Sugar themes, responsive-*, layouts-app-shell, etc. */
```

**After (v5.0.0):**
```css
/* index.css includes ALL ~158 CSS files, parity with maranello.css */
@import 'maranello-luce-design-business/css';
/* Full system: all themes, responsive, runtime layouts included */
```

**Migrate:**
1. No code change needed — your import now delivers the complete system.
2. If bundle size is a concern, switch to selective imports:
   ```css
   @import 'maranello-luce-design-business/css/tokens.css';
   @import 'maranello-luce-design-business/css/base.css';
   @import 'maranello-luce-design-business/css/components.css';
   /* import only the files you need */
   ```

---

### [BC-4] Sugar theme: reduced `!important` specificity overrides

**Why it changed:** 128 `!important` declarations in `themes-sugar-components.css` created
specificity wars, making customization difficult and causing unexpected cascade behavior.
Reduced to 9 (93% reduction) by using higher-specificity selectors instead.

**Who is affected:** Consumers who wrote custom Sugar theme overrides relying on `!important`
to match or beat the design system's specificity.

**Before (v4.20.0):**
```css
/* themes-sugar-components.css used !important extensively: */
body.mn-sugar .mn-card { background: #ffffff !important; }
body.mn-sugar .mn-sidebar { background: #ebedf0 !important; }

/* Your override needed !important to compete: */
body.mn-sugar .mn-card { background: #f0f0f0 !important; }
```

**After (v5.0.0):**
```css
/* Higher-specificity selectors replace most !important: */
body.mn-sugar .mn-card { background: #ffffff; }
body.mn-sugar .mn-sidebar { background: #ebedf0; }

/* Only 9 !important remain (beating JS inline styles): */
body.mn-sugar [style*="background"] .mn-text { color: #111111 !important; }

/* Your override no longer needs !important: */
body.mn-sugar .mn-card { background: #f0f0f0; }
```

**Migrate:**
1. Test your Sugar theme visuals after upgrading.
2. Remove unnecessary `!important` from your Sugar overrides — they likely no longer need it.
3. If a custom override stopped working, increase selector specificity instead of adding
   `!important` back.

## Non-Breaking Changes

- `StateScaffold` still accepts `'partial'` — it now means "degraded content" rather than
  "fully loaded." No runtime error if you keep using it, but the visual treatment may differ.
- `PanelOrchestrator` third argument is optional — existing two-argument usage continues to work.
- New CSS files added to `index.css` are additive — no existing styles were removed or renamed.

## Automated Migration

```bash
# BC-1: Replace partial -> ready where it means "data loaded"
# Review each match — only replace where partial meant "success"
grep -rn "setState('partial')" src/ --include='*.ts' --include='*.js'
# Then manually or with sed:
find src -name '*.ts' -o -name '*.js' | \
  xargs sed -i '' "s/setState('partial')/setState('ready')/g"

# BC-2: Add shell parameter (manual — context-dependent)
grep -rn "new PanelOrchestrator(" src/ --include='*.ts'

# BC-4: Remove unnecessary !important from Sugar overrides
grep -rn '!important' src/css/ --include='*sugar*'
```
